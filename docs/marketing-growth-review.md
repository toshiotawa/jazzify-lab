## 1. 結論

今のJazzifyがやるべきことは、次の3つだけ。**この3つ以外は14日間やらない。**

1. **計測をゼロから最小構成で入れる。** 現状、GA4もイベントログもUTMも一切ない（事実）。「登録74人がどこから来て、何人が初回プレイし、何人が“面白い”まで届いたか」が誰にも分からない状態で、価格・広告・クーポンの議論は全部無意味。最初の1週間で入れる。
2. **SNSで刺さった「パリィ」訴求と初回体験を一致させる。** 「パリィ」はコード内部の開発用語としてのみ存在し、**LP・UI文言のどこにも露出していない**（事実）。12万閲覧を生んだフックと、実際にユーザーが見る画面が完全に乖離している。バズの熱を課金導線に接続できていないのが最大の機会損失。ただしやるのは動画・コピー差し替えとCTA改善まで。新しいデモ体験は作らない。
3. **登録直後に1回、成功させる。** Webは登録後ダッシュボードに放り投げるだけ（事実）。既存チュートリアルへの強制遷移で「初回成功体験」を保証し、`first_success`（後述）を最重要KPIとして追う。

**一旦捨てるもの（14日以内はやらない）**: クーポン / 広告 / ステップメール（トライアルメール3通含む） / 新デモ開発 / 海外展開 / 追加コンテンツ制作 / 管理画面整備。

例外として**メール許諾チェックボックス（同意ログのみ）だけは軽いので14日内に仕込んでよい**。送る仕組みは後でいいが、同意は登録時にしか取れない。詳細は13章。

今のJazzifyに必要なのは完成度ではなく**熱の接続**である。SNSで生まれた熱を、LP → 登録 → 初回成功 → ペイウォールまで1本の線にする。この線が通るまで、他は全部ノイズ。

## 2. 今の最大の問題

**「登録は増えているが有料化が弱い」のではなく、「有料化までの間に何が起きているか観測できていない」のが問題。** その上で、コードから確認できた構造的ボトルネックを仮説として整理する。

### 仮説A: バズの訴求とプロダクトの初回体験が別物（確度: 高）

- SNSで反応したのは「パリィだけで敵を倒すジャズ学習ゲー」= 耳コピバトルのOSMDモード（`chord_osmd`、ハンマーを弾き返す演出）。
- しかしWebの新規ユーザーが最初に見るのは、ダッシュボード上の「メインクエストを始める」CTA（`src/components/dashboard/Dashboard.tsx`）。パリィ体験に最短で到達する導線はない（事実）。
- LPの3モード紹介にも「パリィ」の語はなく、ヒーローコピーは「ジャズピアノを、ゲームのように弾いて覚える。」という汎用訴求（`src/components/landing/landingCopy.ts`）。
- **SNSから来た人は「見た動画と違うもの」に着地している可能性が高い（推測・要検証）。**

### 仮説B: 登録後オンボーディングがWebに存在しない（確度: 高・事実）

- iOSには初回起動時の自動オンボーディング（`OnboardingView.swift`）があるが、**Webは登録完了→ダッシュボード放置**。登録直後にゲームを1回もプレイさせる仕組みがない。
- 登録自体もメール→OTP6桁→ニックネーム→規約同意の最低4ステップ（`AuthLanding.tsx`, `VerifyOtpPage.tsx`, `ProfileWizard.tsx`）。

### 仮説C: 「価値を実感する前にペイウォール、実感した後は何もない」（確度: 中）

- 無料範囲はメインクエスト第1チャプター（`MAIN_QUEST_FREE_MAX_BLOCK_NUMBER = 1`）とサバイバル各マップ先頭ブロックのみ（事実）。
- 範囲自体は狭くないが、**「無料範囲を完走した瞬間」に何が起きるかが設計されていない**。ペイウォール（`WebPaywallModal.tsx`）は汎用文言で、直前の体験（あと少しでクリアだったのに、等）と接続していない。
- リマインダー・ウェルカムメールも未実装（事実。Resendは退会メールのみ、`netlify/functions/deleteAccount.ts`）。トライアルを始めなかった人・7日目の人に何も送っていない。

### 仮説D: MIDIハードルの実態が不明（確度: 不明・最優先で計測すべき）

- タッチ鍵盤・マイク入力のフォールバックは実装済み（事実、`SurvivalGameScreen.tsx` のPIXI鍵盤等）だが、Safari/iOSブラウザにはMIDI非対応警告モーダルが出る（`MidiWarningModal.tsx`）。
- **「登録したがMIDIがなくて離脱」が何%いるかは現状データで一切分からない。**

### 補足: 認証率が高いのは当然

OTP方式なので認証しないとそもそも入れない。認証率88%はファネルの成果ではなく仕組みの帰結。誇るべき数字ではない。

## 3. 最初に狙うべきターゲット

### 主戦場（1つに絞る）

**国内の「ピアノは弾けるがジャズは弾けない」20〜40代。MIDIキーボードを既に持っているか、DTM/ゲーム文化に抵抗がない層。**

理由:
- パリィ動画に反応して415保存したのは「音楽×ゲームのリテラシーがある層」。完全初心者は動画を見ても「自分に弾ける」と思わない（推測）。
- プロダクトの実体（コード進行で敵を倒す、ii-V-I、OSMD譜面パリィ）は、コードネームが読める人が最短で気持ちよくなる設計（事実）。
- 完全初心者向けにはSimply Piano等の月額1,000円前後の巨人がいて、正面衝突は負け筋（競合調査より）。

### 第2候補・第3候補

- 第2: 耳コピ・アドリブに憧れるクラシック出身者（教材はほぼ揃っている）。
- 第3: 英語圏のジャズ学習者（Open Studio/PWJの周辺層）。**ただし今は着手しない**（理由は本レポート「12. 国内 vs 海外」参照）。

### 今は狙わない層

- **完全初心者**: 「鍵盤の場所から」の層はオンボーディングコストが高く、単価3,980円を正当化しにくい。
- **音大生・講師**: 数が少なくB2B的な営業が必要。後で口コミ源として活用すれば良い。
- **海外ユーザー**: 翻訳・決済・コンテンツの穴が多い（12章）。英語LPは維持コストゼロなら放置でよいが、投資はしない。

### SNS反応を受けた訴求変更

変えるべき。現LPの「ジャズピアノを、ゲームのように弾いて覚える。」は正しいが弱い。実際に反応が取れた「パリィだけで敵を倒す」のような**具体的な絵が浮かぶフック**をヒーローに昇格させるべき（詳細は9章）。

## 4. 捨てるべき訴求・機能・導線

### 訴求

- **「異世界ジャズ冒険」トーン**: `SiteFooter.tsx` の「ジャズ異世界で始まる音楽冒険」はLPの実用トーンと分裂している（事実）。ファンタジー世界観はプロダクト内の演出に留め、獲得コピーからは捨てる。
- **「コード、リズム、耳コピ、アドリブを〜」の全部盛り訴求**: 4つ並べると何も残らない。当面「耳とタイミング（パリィ）」に寄せる。
- **英語圏向け獲得施策**: en.jazzify.jpは存在するだけでよい。広告・投稿・コンテンツ翻訳の追加投資は凍結。

### 機能

- **開発者向けコースの作り込み**: 直近マイグレーションは `developer_course_donna_lee_osmd.sql` 等、developer_course系が大量（git statusの未コミット6件も全てコンテンツ調整）。**これは典型的な「作り込みへの逃避」**（9章プロダクト優先順位で詳述）。
- **日記・ミッション・ランキングUIの拡充**: バックエンドは実装済みだがWebシェルから未マウントの可能性（要確認）。ユーザー233人でコミュニティ機能は動かない。凍結。
- **新ゲームモード追加**: モードは既に7種類以上ある。増やすほど初回体験が薄まる。

### 導線

- **LP上の「月額プランを始める」「年額プランを始める」CTA**: どちらも `/signup` に飛ぶだけ（事実）。決済期待で押した人を裏切る。文言を「無料で始める」に統一するか、登録後に該当プランのチェックアウトへ直行させる。

## 5. オンボーディング改善案

### 理想フロー（登録から初回成功体験まで）

```
SNS投稿（パリィ動画）
  → LP: ヒーローで同じパリィ映像 + 「1分だけ体験する」
  → LPデモ（既存の LpDemo / OnboardingExperience をそのまま使う。登録不要）
  → 「無料登録して続きをプレイ」→ /signup
  → OTP認証 → ニックネーム（規約同意はチェック1つに統合）
     + 任意チェック「Jazzifyの使い方・練習課題・無料トライアル案内を受け取る」
       （デフォルトOFF・規約同意と分離。同意ログを保存。詳細は13章）
  → 【新設】登録直後に自動でチュートリアルへリダイレクト
     （既存の /main/play/survival-tutorial または耳コピチュートリアルを流用。新規開発しない）
  → チュートリアル完了 = first_success 記録 → ダッシュボード初回表示
  → メインクエスト第1チャプターへ誘導
```

**LPデモへの注意**: 既存実装の流用で済む範囲（動画差し替え・コピー変更・CTA改善）だけやる。「パリィを1回体験できる新デモ」をここで作り始めると、また開発への逃避になる。ゲームデモの新規開発が必要だと分かった時点で後回しにし、登録後チュートリアル側で成功体験を担保する。

### コードから見つけた離脱ポイントと対処

| 離脱ポイント（事実） | 対処 |
|---|---|
| Web登録後オンボーディングなし（ダッシュボード放置） | `AuthGate`/`VerifyOtpPage` の遷移先を初回のみチュートリアルに変更。既存資産流用でP0 |
| 登録4ステップ+auth初期化5秒タイムアウト（`App.tsx`） | OTP画面の文言改善は後回し。まず計測してどこで落ちるか見る |
| Safari/iOSブラウザのMIDI警告モーダル（`MidiWarningModal.tsx`） | 警告ではなく「タッチ鍵盤で今すぐ試す→アプリ版DLで本格プレイ」の前向き分岐に書き換え |
| 音声はユーザー操作までアンロックされない（`MidiController.ts`） | チュートリアル冒頭に「タップしてスタート」を明示（既存挙動の言語化のみ） |
| ヘッダーにクエスト/サバイバルしかなく耳コピバトルへの直接導線がない（`GameHeader.tsx`） | 初回導線に置くモードを絞る以上、当面はこのままで可。ただしパリィ体験がクエスト第1章の早い位置に来るよう課題順を確認 |

### 初回導線に置くモード / 隠すモード

- **置く**: 耳コピバトル（OSMDパリィ）を最初の成功体験に。次にメインクエスト第1チャプター。
- **見せるが推さない**: サバイバル、コードラン（2週目以降の回遊先）。
- **初回からは隠す**: ファンタジー、ストーリー、風船ラッシュ、精密モード。選択肢が多いほど初回は迷う。

## 6. 料金・無料範囲・トライアル設計

### 現価格の判断

- **月額¥3,980は維持。** 競合調査の結果、ジャズ特化サブスク（Open Studio $39/月、PianoWithJonny $39.95/月）と同水準で、専門サービスとしては妥当。汎用アプリ（flowkey ¥890〜/月）とは比較させない訴求が前提。
- **年額¥34,800も維持だが、初期検証では推さない。** n=4の段階で年額を売ると、解約・返金・LTVの学習が12ヶ月遅れる。当面は月額を主役に、年額はページに残すだけ。
- **値下げはしない。** 有料4人の原因が価格である証拠はゼロ。安売りは後から戻せない。

### 7日間トライアル

- 実装は正しい（Lemon Squeezyのtrial variant + `trial_used` 管理、事実）。設計上の問題は**トライアル中の体験設計とメールが存在しないこと**。
- ただし**トライアルメールの実装は今ではない**。トライアル開始者がほとんど出ていない段階でメールを作っても改善インパクトは小さい。まず「登録→初回成功→ペイウォール到達」の線を通す。メール全般の設計（許諾の取り方・攻略メール・Resendの使い方）は13章にまとめた。トライアル誘導はその攻略メールの3通目として扱い、単独の「トライアルメール3通」は作らない。

### 無料範囲

- 現状（メインクエスト第1チャプター+サバイバル先頭ブロック）は**広さより「終わり方」が問題**。無料最終レッスンのクリア画面に「第2章の1本目をチラ見せ→トライアルCTA」を置く。範囲自体の拡縮は計測後に判断（要検証）。

### 次に検証すべき料金仮説（優先順）

1. 無料範囲完走者のトライアル開始率（現状不明→まず計測）
2. トライアル開始者の7日後継続率（同上）
3. その後にはじめて「月額のみ vs 年額併売」「クーポン有無」をテスト

### 今やってはいけない料金施策

- **常設割引・常時クーポン**: 定価で買った4人への裏切りかつ価格アンカーの毀損。
- **値下げ**: 上記の通り。
- **プラン多段化（Standard/Platinum等の復活）**: DBに旧ランクenumが残っているが（事実）、単一Premiumのまま。選択肢を増やすと転換率は下がる。
- **`public/llms.txt` の「¥4,980/月」表記の放置**: LPは¥3,980。AI検索経由で誤情報が出る。即修正（P0、事実確認済みの不整合）。

## 7. 集計設計

### 現状（事実）

- 第三者アナリティクス: **なし**（GA4/Mixpanel/PostHog等ゼロ。`package.json`・`index.html` 確認済み）
- UTM/リファラ/first_touch: **なし**（`utm_` はコードベース全体で0件）
- 使えるもの: `profiles`（created_at, signup_platform, country）、`subscriptions`（status, trial_used_at, plan_code）、`subscription_events`、各モードのクリアテーブル（`survival_stage_clears` 等）
- 欠けているもの: 流入元、LP行動、登録開始、初回プレイの統一タイムスタンプ、ペイウォール表示/クリック、オンボーディング完了のサーバー記録

### GA4とSupabaseの役割分担（重要: GAを全部の正本にしない）

GA4は導入していいが、**「どの入口が強いかを見る道具」に限定し、登録後のプロダクト内行動・課金状態の正本にはしない。** 全部をGAに寄せようとすると、また基盤が重くなる。

| 役割 | 使うもの |
|---|---|
| 流入元・LP・SNS投稿別の登録効率 | **GA4** |
| 登録後の初回プレイ・初回成功・ペイウォール・課金状態 | **Supabase / 自前`analytics_events`（正本）** |
| 広告を回す時の最適化 | **GA4 key event + Google Ads連携（P2以降）** |

**GAに入れていいもの**: `page_view`（LP閲覧）、`sign_up_click`、`sign_up`、`tutorial_begin`、`tutorial_complete`、`begin_checkout`、`purchase`。可能な限りGA4の**推奨イベント名**（`sign_up`, `tutorial_begin`, `tutorial_complete`, `begin_checkout`, `purchase`）に寄せる。ただし `first_success` のようなJazzify固有の重要イベントは無理にGAへ寄せず、自前イベントで持つ。

**GAに入れてはいけないもの**（Supabase側で見る）: どの課題をクリアしたか、ミス回数、MIDI接続状態、パリィ成功率、無料範囲完走、ペイウォール表示元、`subscription_status`、`trial_used_at`、paid/canceled、ユーザー単位の詳細行動履歴。理由は単純で、GAはマーケ寄りの集計ツールであり、プロダクト内部の正確な行動DBではないため。

**GA key event（最初はこの3つだけ）**: `sign_up`（無料登録）、`tutorial_complete`（初回体験完了）、`begin_checkout`（トライアル/決済導線に進んだ）。`purchase` は重要だが件数が少ないうちはノイズが大きく、広告最適化の対象にするには早い。key event化はP2（広告テスト着手時）でよい。

**実装方針（最小構成）**:
- `gtag.js` を設置するだけで十分。GTMは今は入れない（タグ管理の複雑さを増やす段階ではない）。
- LP着地時にUTM+`referrer`に加えて**`ga_client_id`もlocalStorageへ保存**し、`profiles.first_touch` にも書き込む。これでGA側の数字とSupabase側の数字を後で照合できる。
- 購入確定イベントはフロントからGAへ送るとズレるため、**正本はLemon Squeezy webhook / Supabaseの確定データ**とする。GAへの`purchase`送信は補助扱いに留め、集計判断には使わない。

**見るべきGAファネル**: `LP view → sign_up_click → sign_up → tutorial_begin → tutorial_complete → begin_checkout`。ここで一番見たいのは購入ではなく、**登録後に初回チュートリアルまで行っているか**（= 7章後半の「第1段階KPI」と同じ問い）。`sign_up→tutorial_begin`が低ければ登録後導線が死んでいる、`tutorial_begin→tutorial_complete`が低ければ初回体験が難しい、`tutorial_complete→begin_checkout`が低ければ価値のピークとペイウォールが繋がっていない、と読む。

**プライバシー・同意まわり（事実確認: 現状GA未導入なので該当ポリシー記載なし）**: GA導入時はプライバシーポリシーにGA利用を明記する（P0の軽微タスク）。EEA等の同意管理が厳しい地域への広告・本格展開時はConsent Mode v2の検討が必要だが、**今の国内検証段階でCMPを作り込むのは過剰**。プライバシーポリシー整備と最低限の計測のみで十分。

### 最小実装（Supabase側。これ以上は作らない）

**1. `analytics_events` テーブル1本**（新規マイグレーション。GA4とは別に、正本としてSupabase側にも同じ節目を残す）

```sql
create table analytics_events (
  id bigint generated always as identity primary key,
  anonymous_id text,                -- LP来訪時にlocalStorageで発番
  user_id uuid references auth.users(id),
  event_name text not null,
  properties jsonb default '{}',
  created_at timestamptz default now()
);
-- RLS: insertはanon/authenticated可、selectはservice_roleのみ
```

**2. profilesに流入元カラム追加**

```sql
alter table profiles add column first_touch jsonb;  -- {utm_source, utm_medium, utm_campaign, referrer, landing_path, ga_client_id, captured_at}
```

**3. イベントは必須7個**（送信ヘルパー `src/utils/track.ts` を新設、fire-and-forgetでゲームループ外のみから呼ぶ）。このうち一部はGA4側にも対応する推奨イベント名で並行送信する（同じ呼び出し箇所から2回fire-and-forgetするだけで、二重の基盤を作るわけではない）。

| 優先 | event_name（Supabase） | 対応するGA4イベント | 発火箇所 |
|---|---|---|---|
| 必須 | `lp_view` | `page_view` | LandingPage マウント時 |
| 必須 | `signup_start` | `sign_up_click` | AuthLanding でメール送信時 |
| 必須 | `signup_complete` | `sign_up` | プロフィール作成完了（profiles.created_atと重複するが、anonymous_id連結のため必要） |
| 必須 | `first_play_start` | `tutorial_begin` | 各ゲーム画面の初回開始（profiles.first_play_atも同時更新、冪等） |
| 必須 | `first_success` | `tutorial_complete` | 下記参照。ユーザーごとに初回のみ |
| 必須 | `free_tier_wall_view` | （GAには送らない） | ペイウォール表示（表示元モードをproperties.sourceに） |
| 必須 | `checkout_click` | `begin_checkout` | WebPaywallModal / PricingTable のCTA |
| 任意 | `demo_start` / `demo_complete` | LpDemo / OnboardingExperience。**既存デモに数行足すだけで済む場合のみ**入れる |
| 送らない | `trial_start` / `paid` | 既に `subscriptions` / webhook で取れる。二重管理しない |

**`first_success` の定義（最重要イベント）**: `first_play_start` だけでは「起動したが意味が分からず離脱」と「面白さまで届いた」を区別できない。**初回で気持ちよく成功した瞬間**を1イベントで捕まえる。定義は次のいずれか早い方（properties.kindで区別）:

- チュートリアル完了（登録直後の強制遷移先）
- 初回ステージ/課題クリア
- パリィ成功（耳コピバトル系に入った場合）

送信箇所はチュートリアル完了ハンドラと各モードのクリア処理（`upsertSurvivalStageClear` 等の呼び出し元）。ゲームループ内ではなくクリア確定イベントで1回だけ送る。

**4. 集計はSQL 1本 + Supabase SQL Editorで週次確認**（ダッシュボード製品は入れない）

```sql
-- 週次ファネル（utm_source別）
with ev as (
  select user_id, event_name, min(created_at) as first_at
  from analytics_events
  where user_id is not null
  group by 1, 2
)
select
  coalesce(p.first_touch->>'utm_source', 'organic') as source,
  count(*) as signups,
  count(p.first_play_at) as first_played,
  count(*) filter (where exists (
    select 1 from ev where ev.user_id = p.id and ev.event_name = 'first_success'
  )) as first_success,
  count(*) filter (where exists (
    select 1 from ev where ev.user_id = p.id and ev.event_name = 'free_tier_wall_view'
  )) as wall_reached,
  count(s.trial_used_at) as trials,
  count(*) filter (where s.entitlement_state = 'active' and s.status <> 'trial') as paid
from profiles p
left join subscriptions s on s.user_id = p.id
where p.created_at >= now() - interval '7 days'
group by 1 order by 2 desc;
```

**5. 週次で見る数字は6個だけ**: 登録数 / 初回プレイ率 / **初回成功率** / ペイウォール到達数 / トライアル開始率 / 有料転換数。

### KPIの置き方: 段階で分ける（トライアル開始率はまだ主役ではない）

いきなり「トライアル開始率」を唯一のKPIに置くのは早い。手前が壊れていたら下流の数字に意味がないため、**段階制**にする。

- **第1段階（今ここ）: 登録→初回成功率。** これが低いなら、ペイウォールも価格も関係ない。体験まで届いていないだけ。目安50%未満ならオンボーディングを直すこと以外やらない。
- **第2段階: 初回成功→ペイウォール到達率。** 低いなら無料導線が散らかっているか、価値のピークにペイウォールが置けていない。
- **第3段階: ペイウォール→トライアル開始率。** ここで初めて価格・文言・無料範囲の議論をする。

**今の最重要KPIは「登録者のうち何%が `first_success` まで行ったか」。**

### SNS投稿単位のトラッキング

投稿ごとに `https://jazzify.jp/?utm_source=x&utm_campaign=parry01` のようなUTM付きリンクを使うだけで、上記実装でLP→登録→有料まで連結できる。**Xのプロフィールリンク・固定ポスト・各投稿のリンクを全てUTM付きに差し替えること**（コード変更不要、運用）。

## 8. クーポン設計

### 結論: **14日以内はクーポンを一切やらない**

まだ価格が問題かどうか分かっていない。この段階でクーポンを出すと、「安いから入った人」と「価値を感じて入った人」が混ざり、有料転換の学習データが濁る。テスターへのお礼も、クーポンではなくまず個別の御礼メッセージとヒアリングで十分。クーポンに着手するのは、段階KPI（7章）の第3段階=ペイウォール→トライアル開始率を議論できるデータが貯まってから。以下は**その時のための設計メモ**として残す。

### 現状（事実）

クーポン機能は**未実装**。コード・DB・管理画面のいずれにもなく、`LEMONSQUEEZY_SETUP.md` でLemonのDiscounts機能は「OFF（未使用）」と明記。

### 方針（着手時）: 自前実装しない。Lemon Squeezyのネイティブ割引コードを使う

Lemonのダッシュボードでdiscount codeを発行し、チェックアウトURLに `checkout[discount_code]` を渡すだけで済む。改修は `netlify/functions/lemonsqueezyResolveLink.ts` にクエリパラメータを1つ通す程度。iOS側はApp Store Connectのオファーコードで別管理（当面やらない）。

### 3種類の設計

| 対象 | 設計 | 理由 |
|---|---|---|
| **テスター向けお礼** | 個別コード、**初回3ヶ月30%オフ**、有効期限30日、人数上限=テスター数 | 「永久割引」はLTV計算を壊す。感謝は深く・期間は短く |

- コード利用数 / 発行数（消化率）
- クーポン経由トライアル→有料転換率 vs 定価組の転換率（**定価組より著しく低ければクーポンは客層を汚している**）
- 3ヶ月後継続率の定価組との差

## 9. 広告・SNSテスト案

### 広告を今回すべきか → **回さない（14日以内は）**

理由: (1) 計測がないので広告の学習が資産にならない、(2) オーガニックで12万閲覧→74登録が取れており、まずこの導線の転換率を上げる方がROIが高い、(3) n=4では「何を最適化する広告か」すら決められない。広告は計測実装+オンボーディング改善後、**登録〜初回プレイをCVとする少額テスト**（有料会員獲得を直接狙わない）から。

### 最初に出すのは2本。3本目は条件付き（オーガニック投稿として。当たれば同素材を広告化）

最初から3本出すと、何が効いたか分からなくなる。まず1本目・2本目を順に出し、**3本目（MIDIハードル打ち消し）は計測で初回プレイ率が低かった場合にだけ出す**。

**1本目: 「パリィ強化版」（継続・本命）**
- 内容: OSMDパリィで敵のハンマーを弾き返す実プレイ画面録画15〜30秒。字幕「ジャズの譜面、ゲームなら読める」
- 狙い: 既に当たったフックの再現性確認と、UTM付きリンクでの転換計測
- 仮説: 保存する層はピアノ経験者。プロフィール→LP遷移率が改善すれば登録も伸びる
- 成功判定: 投稿リンク経由の登録20人以上/本、登録→初回プレイ率50%以上

**2本目: 「Before/After」（経験者向け獲得訴求）**
- 内容: 「クラシックは弾けるのにジャズは弾けない人へ」→ ii-V-Iをコードランで弾けるようになる過程を20秒で
- 狙い: バズ用（1本目）と獲得用の分離検証。刺さる層の特定
- 仮説: 「自分ごと化」コピーの方が保存よりリンククリックが増える
- 成功判定: リンクCTR率が1本目比で高い、かつ登録後のトライアル開始が1人でも出る

**3本目（条件付き・初回プレイ率が低かった場合のみ）: 「MIDIキーボードを繋ぐだけ」（ハードル打ち消し）**
- 内容: 手元+画面の2画角。安いMIDIキーボードを繋いで60秒でプレイ開始する様子。「61鍵1万円で始まる」
- 狙い: 機材ハードル仮説（仮説D）を訴求で打ち消せるかの検証
- 仮説: 「機材が要る=無理」と思っている潜在層の登録が増える
- 成功判定: この投稿経由の登録者の初回プレイ率が他投稿と同等以上（機材が本当に障壁なら低く出る→事実が取れる）

### SNS導線の改善（コード変更ほぼ不要の即日タスク）

- 「パリィだけで敵を倒すジャズ学習ゲー」訴求は**継続**。ただしリンク先LPのヒーローに同じ映像を置いて一致させる。
- 初心者向けより**経験者向け訴求を優先**（3章の通り）。
- バズ用と獲得用は**分ける**。バズ投稿の固定リプライに獲得導線（UTM付きLP+「無料で1分デモ」）を置く。
- プロフィール・固定ポスト・リンクを全てUTM付きに統一。
- **`SiteFooter.tsx` のSNSリンクが `href="#"` のまま**（事実）。逆流導線が死んでいる。即修正。

- **売るために必要**: 計測、登録直後の1回の成功体験、ペイウォールの文脈化、SNS導線の整備、メール許諾の同意ログ（取得だけ）。以上。（ステップメール・クーポンは「売るために必要」に見えるが、母数が出るまでは効かない。P1/P2のトリガー待ち）
- **作り込みへの逃避と判断する根拠（事実）**: 直近50コミットとgit statusの未コミット6ファイルは、Donna Lee OSMD・精密モードの判定調整・developer_course系コンテンツに集中している。`SurvivalGameScreen.tsx` は7,146行、耳コピ系はWeb+iOS二重実装で計6万行超。一方、ウェルカムメール0通・計測0行・Webオンボーディング0画面。**プロダクトの完成度と商売の完成度が極端に非対称。** 判定精度±250msの統一より、登録者74人のうち何人がプレイしたかを知る方が今は100倍重要。

## 12. 国内 vs 海外・競合・翻訳状況（補論）

### 競合調査の結論: 「ジャズ特化 × ゲーム × MIDI」はほぼ空白市場

外部調査（2026年7月時点、出典は調査エージェントのレポート参照）:

| サービス | 月額目安 | ジャズ特化 | ゲーム性 | MIDI |
|---|---|---|---|---|
| Melodics | Standard $10.99〜（上位プランあり） | △ | ◎ | ◎ |
| Yousician / Simply Piano / flowkey | $10–20（国内flowkey ¥890〜） | △ | ◎ | △〜○ |
| Open Studio | $39（年払い$33） | ◎ | ✕（動画） | ✕ |
| PianoWithJonny | $39.95 | ◎ | △ | ✕ |

- 3軸を同時に満たす直接競合は国内外とも見当たらない。Melodics（汎用リズム訓練）とOpen Studio（ジャズ動画）の間の空白。
- 価格ポジションの注意: **ゲーミフィケーション系アプリ（Melodics Standard $10.99、flowkey ¥890〜）と比較されたら¥3,980は2〜4倍で確実に負ける。** 正当化できるのはジャズ専門サブスク（Open Studio $39、PWJ $39.95）と比較された場合のみ。つまりJazzifyは価格で戦うのではなく、**「ジャズ特化・譜面・耳コピ・アドリブ」で別カテゴリに置く訴求が生命線**。LP・投稿・広告で比較軸を常にジャズ専門側に固定すること。

### 国内か海外か → **国内先行。海外は「準備だけ維持、投資ゼロ」**

英語圏の市場は大きい（オンライン音楽教育 $4.6B、PWJ会員1万+等）が、コードベース調査で判明した海外展開の実態は以下（事実）:

- **UIシェルはかなり英語化済み**: `shouldUseEnglishCopy` が約82ファイルで参照され、ダッシュボード・クエスト・サバイバル・耳コピ・課金UIは日英分岐あり。これは資産。
- **しかし穴が多い**:
  - DBコンテンツは `title_en` 等の二言語スキーマだが、充足率は不明。耳コピ系は英語欠落時に**空文字表示**（`lessonCopy.ts`、日本語フォールバックなし）。
  - ヘルプ全ページ・マイページ（`MypageModal.tsx`）・特商法・MIDI購入ガイド（iOS）は日本語固定。
  - お知らせは `publish_en=false` がデフォルトで、英語ユーザーには**空欄**。
  - ファンタジーストーリーは日本語のみ（英語UIではボタン非表示）。
  - **決済はJPY固定**（Lemon）。英語LPのUSDは参考換算表示のみ。米国ユーザーはJPY請求+為替手数料を受ける。
- つまり「英語ユーザーが今日サインアップすると、コアゲームは遊べるが、教材本文・サポート・請求で日本のサービスだと露呈する」状態。

**判断**: 翻訳の骨格ができているのは強みだが、海外で戦うにはコンテンツ`_en`充足・USD決済（Lemonのマルチ通貨 or 別store）・英語サポート導線が必要で、これは今の4人の有料会員とn=233では時期尚早。**国内で「初回体験→トライアル→有料」の勝ちパターンを数字で確立してから、同じファネルを英語圏に複製する。** 目安として、国内で月次トライアル開始20件・トライアル→有料転換30%が見えたら海外Phase開始を検討。

### 海外向けに今だけやっておくこと（工数ほぼゼロ）

- en.jazzify.jp とApp Store英語ページは現状維持（削らない）。
- 英語圏からのオーガニック登録も7章の計測に乗るので、`profiles.country` / `first_touch` で海外流入比率だけモニタリング（要検証データとして貯める)。
- 海外ユーザーから問い合わせが来たら個別対応し、需要の質を記録する。

## 13. メール設計（補論: 許諾・攻略メール・Resendの使い方）

### 位置づけ

メールは主役ではない。**主役は「登録直後にアプリ内でパリィ体験へ飛ばす」導線（5章）で、メールはそこから漏れた人を拾う補助輪。** ただし、許諾だけは登録時にしか取れないため、チェックボックスと同意ログの仕込みだけ先行させる価値がある（P0.5）。

### 無料登録時のメール許諾チェックボックス

設計原則:

- **デフォルトOFF。** 利用規約同意とは分離し、登録必須にも同意済み風にも見せない。チェックしなくても登録できることを明記。
- トライアル誘導メールは実質的に広告宣伝メールであり、日本の特定電子メール法は**オプトイン規制**（事前承諾のない広告メールは原則禁止）。送信時には送信者名・配信停止URL・配信停止できる旨・住所・問い合わせ先の**表示義務**もある。雑にやらない。
- 文言は「メルマガ」ではなく**ユーザーの利益が見える案内**にする。

チェックボックス文言案:

```text
□ 3日間のJazzify攻略メールを受け取る
  パリィ体験の始め方、最初にやるべき課題、無料トライアルの案内をお送りします。
  ※いつでも配信停止できます。チェックしなくても無料登録できます。
```

同意ログ（新規migration、`profiles` に追加）:

```sql
alter table profiles add column marketing_email_opt_in boolean default false;
alter table profiles add column marketing_email_opt_in_at timestamptz;
alter table profiles add column marketing_email_opt_in_source text;   -- 'signup_wizard' 等
alter table profiles add column marketing_email_opt_in_text text;     -- 同意時に表示していた文言（証跡）
```

実装箇所: `src/components/auth/ProfileWizard.tsx`（チェックUI）+ `src/stores/authStore.ts` の `createProfile`（保存）。

### 送るのは「トライアル誘導メール」ではなく「攻略メール」

目的はメールで売ることではなく、**登録者を初回成功まで戻すこと**。その結果としてトライアルに誘導する。売り込み色を前に出すと解除される。

最初は**1通だけ**。ステップメールを最初から組まない。

**1通目（登録直後）— 目的: 初回プレイ誘導**

```text
件名: まずは1分、敵のフレーズをパリィしてみてください

Jazzifyへようこそ。
まずは説明を読むより、1分だけプレイしてみてください。

こちらから最初のパリィ課題を開けます。
[パリィ課題を開く]

MIDIキーボードがなくても、まずはタッチ鍵盤で試せます。
```

1通目が回ってから追加する2通（P2）:

- **2通目（翌日）— 価値の言語化**: 件名「ジャズが難しい理由は、理論不足より“反応速度”です」。理論を増やす前に耳で聴いて指で返す、Jazzifyはそれをゲーム化している、無料範囲でここまで試せる。CTA「続きをプレイする」
- **3通目（3日後）— ここで初めてトライアル誘導**: 件名「第2章以降を7日間無料で試せます」。無料範囲の続き、Premiumで解放されるもの、7日間無料、いつでもキャンセル可。CTA「無料トライアルを始める」

さらに先（P2の後半）は行動別分岐: 登録済み未プレイ→「まず1分」、初回プレイ済み成功なし→「タッチ鍵盤でもOK」、初回成功済みペイウォール未到達→「次はこの課題」、ペイウォール表示済み未トライアル→「7日間無料で第2章へ」。**ただし最初からここまで作ると重い。全員共通の1〜3通が回り、`analytics_events` でセグメント母数が見えてから。**

### Resendの使い方: APIで送るだけ。管理画面運用にしない

| 方式 | 判断 |
|---|---|
| Resend Broadcasts（画面で書いて一斉配信） | **やらない**。ニュースレター運用になり、今の用途に重い |
| Resend Automations（イベントトリガーのステップメール） | **後回し**。今やるとCRM設計に逃げる |
| **Jazzify側コードからResend APIで送信** | **これでいく**。文章・タイミングはコード側、Resendは送信だけ |

具体的には、プロフィール作成完了（許諾あり）をトリガーに、Netlify FunctionからResend APIで1通送る。既存の `netlify/functions/deleteAccount.ts` にResend利用の先例があるので構成はそれに倣う。

### 実装の順番（再掲）

1. UTM・計測（P0）
2. 登録直後チュートリアル誘導（P0）
3. メール許諾チェックボックス+同意ログ（P0.5。送信機能なしで先行可）
4. 登録直後の案内メール1通（P1）
5. 攻略メール3通化 → 行動別分岐（P2）

チェックボックスだけ先に作っても送る中身がなければ効果はゼロだが、**同意ログを取り始める価値はある**。逆に、同意を取らずに後からメールを送ることはできない。
